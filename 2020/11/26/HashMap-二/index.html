Overview

HashMap(一) 中简单介绍了直接寻址法和散列寻址法，并介绍了链接法解决散列冲突的散列表，和HashMap中在某些条件下会使用了红黑树作为散列表的槽位。这篇博客介绍HashMap中的数据结构和3个的动态集合操作，insert，delete，search

Table

关于散列表，HashMap中使用一个Node数组维护散列表结构，数组中的每一个位置在HashMap中被称为bin或者bucket

结点结构

Node和TreeNode是HashMap中使用的基本存储结构，如下图

Node<K,V>

我们知道Map接口中，有一个内部接口叫Entry,表示映射当中的条目。Node类除了实现了接口Entry之外，内部还有四个成员变量

  • hash 保存着key的散列值，
  • key
  • value
  • next next变量使得Node可以成为一个单链表

回看一下，算法导论中提到的链接法解决散列冲突的散列表的删除算法，

算法的输入是散列表和要被删除的结点。在算法导论中，作者建议使用双链表作为散列表的槽，其理由是认为如果使用双链表，删除算法的时间复杂度就是$O(1)$,只需要将x前驱的后继改成x的后继，x后继的前驱改成x的前驱就行了，从而避免单链表的搜索。但是对于HashMap的使用者来说，删除结点只是输入泛型类型K,并不直接输入结点，依旧需要在链表中搜索对应结点，个人认为这是HashMap使用单链表的原因之一。

TreeNode<K,V>

TreeNode直接继承LinkedHashMap.Entry间接继承自HashMap.Node . LinkedHashMap.Entry中有两个变量before，after因此可以作为常规结点扩展，也可以作为链式结点扩展。

TreeNode包含了红黑树结点结构中的所有元素，

  • parent指向父结点或者null。只有在一颗红黑树中只有根结点才回出现parent为null的情况
  • left指向左孩子或者null
  • right指向右孩子或者null
  • red表明结点是否为红色。

红黑树的高度h和结点数n满足$h≤2\ lg\ (n+1)$，正如HashMap(一)中所说红黑树的基本操作的最坏情形复杂度为$O(lg\ n)$这能够在散列表处于糟糕的散列分配的情况下，提供比链表更快的操作。

红黑树任意一条从根结点到叶子结点的简单路径长度不可能大于另一条的两倍。

动态集合操作

在对于一个map来说，get,put，remove是HashMap基本的方法，其他诸如getOrDefault，putAll 等方法(但不是所有方法)直接或间接的依赖get,put，remove方法。getNode ，putVal ，removeNode这三个方法是Hashmap内部的方法，不可被重写，具有包访问权限。

查看HashMap的源代码，可以看到get,put,delete 基本上只是调用了getNode ，putVal ，removeNode三个内部方法，并不会直接操作散列表的结构。getNode ，putVal ，removeNode方法是HashMap内部散列表的动态集合操作insert,delete,search。

🍀 我们可以看到上面三个方法内部都调用hash(key)，这是HashMap使用的散列函数，这个散列函数使得我们可以通过散列表的大小n和计算出来的散列值hash通过计算得到对应的散列桶的位置其计算方式如下
$$
(n-1)\ &\ hash
$$
将n-1和hash进行按位与。其原理不在这篇博客的主旨中(更关键的是，也没研究过这个，可能在后续博客专门写)。顺带提一下，HashMap使用2的非负整数次幂作为散列表的大小，n-1就是一个梅森数(不一定是梅森素数)，转成2进值就是$log_2{n}个1$

HashMap内部实现了红黑树的结构和相关算法，内容复杂，插入删除时需要维护红黑性质，这会在HashMap(三)单独分析。

getNode

getNode 方法能辅助HashMap中的get或者其他方法从散列表中获取一个结点。

getNode 首先找到散列桶中的第一个元素first，如果存在则继续操作，不存在就直接返回null。找到first后，首先检查第一个结点是否为要找的结点，如果是的话直接返回第一个结点。如果不是，那么根据结点类型执行搜索操作。如果是链式结点，那么做单链表的搜索，如果是红黑树则做红黑树的搜索。

🍀 getNode,putVal,removeNode 类似的都会检查散列桶的第一个结点。笔者的看法是，不管结点是链式结点还是红黑树结点，都是HashMap为了解决散列冲突使用的结构，然而一个好的设计不管是HashMap的内部结构还是程序员自己写的hashCode方法都应该减少冲突的发生，因此总是检查第一个结点可以避免后续的复杂操作。

removeNode

getNode 方法能辅助HashMap中的get或者其他方法从散列表中删除一个结点。removeNode可以和getNode结合看，因为要删除一个结点必须先找到那个结点，removeNode的前半部分和getNode类似，后半部分根据条件的不同决定是否删除结点。

如果matchvalue 为true ，则仅当结点的value和参数value相等时才会发生删除。

如果movable 为false 那么在移除结点时不会移除其他结点。

可以看到在24行之前，removeNode类似于geiNode都在寻找要删除的结点。主要的一些不同的是

  • 找到结点后让node指向结点而不是return
  • 记录了散列桶的位置，因为删除结点可能会删除散列桶的第一个结点，这个时候就需要改变散列表中的指向的根结点。

如果node的值满足移除的条件则继续移除操作，此时可以分为3种情况

  • 情况一 结点是红黑树结点

    ​ 直接调用红黑树的删除算法 ，这里也会有散列桶的位置时红黑树的根结点的情况，在算法内部解决的这个问题

  • 情况二 结点是链式结点，并且是头结点

    ​ 将散列表对应的散列桶位置指向node的下一个结点

  • 情况三 结点是链式结点，并且不是头结点

    ​ 使用单链表的删除

然后就是修改 ++modCount，把记录的散列表大小减一，调用给子类的方法 afterNodeRemoval(node) 并返回被删除的结点

putVal

putVal方法能够辅助HashMap中的put方法及其他方法向散列表中插入一个结点，其中hash是key的散列值，key，value不做解释，onlyIfAbsent 用来实现若没有则添加的语义，如果为true，那么只有在key没有映射值或者映射了null才会执行添加操作，evict是为HashMap子类的设计的。

putVal 在执行插入前会检查散列表，如果表为空或者表的大小等于0，就会通过resize()方法给出一个合适大小的散列桶。

接下来的操作可以基本可以分为两种情况

  • 情况一：hash对应的散列桶不存在

    此时只需要在散列表对应位置存放一个链式结点就行(对应的散列桶不存在就可以认为key没有映射到value,符合onlyIfAbsent语义)

  • 情况二：hash对应的结点p存在

    找到对应key的结点，根据onlyIfAbsent 决定是否插入值。找到结点可以分三种小情况，此时定义引用e代表搜索到的结点

    • p就是对应key的结点

    • p不是对应key的结点，但是p是红黑树结点

      此时在红黑树中搜索对应key的结点，找到了e赋值为对应key的结点，否则执行红黑树的插入

    • p不是对应key的结点，是链式结点

      此时链式搜索对应key的结点，如果找到，e赋值为对应key结点的，如果找不到就只执行插入，执行插入的时候检查是否需要将链表树化

如果没有找到旧的Node,就会走到代码的最后几行，自增modCount，如果大小超过限度值threshold，则进行resize操作，此时返回
null。

🍀 resize()能够根据当前散列表的情况给出一个合适的散列表结构，可能是由于散列表中元素过多，或者是散列表还没有初始化

afterNodeAccess(e)，afterNodeInsertion(evict) 是给子类提供的方法，HashMap自身不做操作。
